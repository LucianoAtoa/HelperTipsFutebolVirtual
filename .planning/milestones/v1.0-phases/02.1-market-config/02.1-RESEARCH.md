## Summary

Esta fase adiciona um sistema de entradas complementares sobre a infraestrutura já existente da Phase 2. O trabalho divide-se em três eixos independentes: (1) extensão do schema PostgreSQL com duas novas tabelas; (2) nova lógica de negócio em Python puro para cálculo de stakes Martingale e validação de complementares por placar; (3) expansão do dashboard Dash com um novo card de ROI Complementares com breakdown por mercado.

O código existente foi construído com padrões que facilitam a extensão: `ensure_schema()` em `db.py` usa `CREATE TABLE IF NOT EXISTS` e `ALTER TABLE ADD COLUMN IF NOT EXISTS`, `calculate_roi()` em `queries.py` é pura Python sem dependência de banco, e o master callback do dashboard aceita múltiplos Outputs. Nenhum padrão estabelecido precisa ser quebrado — a fase é aditiva.

O único ponto de atenção é a representação de `regra_validacao` na tabela `complementares`. A decisão de como armazenar a regra (string de expressão Python-like, enum de valores, ou código avaliável) afeta a testabilidade e a segurança. A recomendação abaixo usa um enum cuidadosamente definido, que é a abordagem mais segura e mais testável para este caso.

**Primary recommendation:** Implementar `regra_validacao` como um enum string (ex: `"over_3_5"`, `"over_5_plus"`) mapeado para funções Python em um dict, evitando `eval()` e mantendo validação explícita. Seed data em SQL puro para `ensure_schema()`.

### Pattern 1: Extensão de ensure_schema() com seed data

**What:** Adicionar CREATE TABLE + INSERT ON CONFLICT DO NOTHING dentro de `ensure_schema()`, o mesmo padrão já usado para `signals` e `parse_failures`.
**When to use:** Sempre que uma nova tabela precisa de dados iniciais que fazem parte da definição do sistema (não dados de usuário).
**Example:**
```python
# Em db.py — dentro do bloco cur.execute() de ensure_schema()
```

### Pattern 2: regra_validacao como enum-string + dispatch dict

**What:** Armazenar `regra_validacao` como um slug string (ex: `"over_3_5"`) e mapear para funções Python validadas em um dict. Nunca usar `eval()`.
**When to use:** Qualquer vez que lógica de negócio variável precisa ser armazenada em banco mas executada em Python.
**Example:**
```python
# Em queries.py (ou novo arquivo market_rules.py)
from typing import Callable

def _parse_placar(placar: str) -> tuple[int, int] | None:
    """Parse 'X-Y' string para (casa, fora) inteiros. Retorna None se inválido."""
    if not placar:
        return None
    try:
        parts = placar.split("-")
        if len(parts) != 2:
            return None
        return int(parts[0]), int(parts[1])
    except (ValueError, AttributeError):
        return None


_REGRA_VALIDATORS: dict[str, Callable[[int, int], bool]] = {
    "over_3_5":         lambda casa, fora: (casa + fora) > 3.5,
    "over_5_plus":      lambda casa, fora: (casa + fora) >= 6,
    "empate_3_3_4_4":   lambda casa, fora: (casa + fora) in (6, 8) and casa == fora,
    "gols_casa_4":      lambda casa, fora: casa == 4,
    "gols_fora_4":      lambda casa, fora: fora == 4,
    "gols_casa_5_plus": lambda casa, fora: casa >= 5,
    "gols_fora_5_plus": lambda casa, fora: fora >= 5,
}


def validar_complementar(regra: str, placar: str | None, resultado_principal: str | None) -> str | None:
    """
    Determina o resultado de uma entrada complementar dado o placar do sinal principal.

    Returns:
        "GREEN" | "RED" | None (pendente)
    """
    # Sinal pendente → complementar pendente
    if resultado_principal is None:
        return None

    # Sinal RED sem placar → todas complementares RED (D-08)
    if resultado_principal == "RED" or placar is None:
        return "RED"

    parsed = _parse_placar(placar)
    if parsed is None:
        return "RED"  # placar malformado — conservador

    casa, fora = parsed
    validator = _REGRA_VALIDATORS.get(regra)
    if validator is None:
        return "RED"  # regra desconhecida — conservador

    return "GREEN" if validator(casa, fora) else "RED"
```

**Nota sobre empate_3_3_4_4:** A regra `(casa + fora) in (6, 8) and casa == fora` cobre exatamente 3-3 (total=6, casa==fora) e 4-4 (total=8, casa==fora), excluindo 2-4/4-2 e similares. Verifica-se que não há falso-positivo: 1-5 total=6 mas casa≠fora, 2-6 total=8 mas casa≠fora.

### Pattern 4: get_complementares_config(conn) — query do banco

**What:** Função de query que carrega a config de complementares de um mercado específico. Segue o padrão de `get_filtered_stats()` — recebe conn, fecha cursor, retorna lista de dicts.

```python
def get_complementares_config(conn, mercado_slug: str) -> list[dict]:
    """
    Retorna lista de complementares ativas para um mercado principal.
    Cada dict tem: id, slug, nome_display, percentual, odd_ref, regra_validacao.
    """
    sql = """
        SELECT c.id, c.slug, c.nome_display, c.percentual, c.odd_ref, c.regra_validacao
        FROM complementares c
        JOIN mercados m ON c.mercado_id = m.id
        WHERE m.slug = %s AND m.ativo = TRUE
        ORDER BY c.percentual DESC, c.slug
    """
    with conn.cursor() as cur:
        cur.execute(sql, (mercado_slug,))
        columns = [desc[0] for desc in cur.description]
        rows = cur.fetchall()
    return [dict(zip(columns, row)) for row in rows]
```

### Anti-Patterns to Avoid

- **eval() para regra_validacao:** Nunca avaliar expressões do banco com `eval()`. Usar dispatch dict como mostrado no Pattern 2.
- **Consultar complementares fora de ensure_schema():** O seed data deve ficar em `ensure_schema()` usando `ON CONFLICT DO NOTHING` — idempotente em qualquer restart.
- **Criar processo separado para complementares:** As complementares são calculadas em tempo de leitura no dashboard (sem listener). Não criar thread/processo adicional.
- **Módulo-level DB connection:** Manter o padrão estabelecido — conn por invocação de callback, fechado em `finally`.
- **Acoplamento entre calculate_roi_complementares e Dash:** A função deve ser pura Python sem imports de Dash, exatamente como `calculate_roi()` existente.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Validação de regras de mercado | Parser de expressão customizado ou DSL | Dict de lambdas (`_REGRA_VALIDATORS`) | 7 regras fixas não justificam um parser; lambdas são testáveis diretamente |
| Seed data em banco | Script de migração separado ou ORM migration | `INSERT ... ON CONFLICT DO NOTHING` em `ensure_schema()` | Padrão já estabelecido no projeto; idempotente |
| Lógica de stake Martingale | Recalcular inline no callback | `calcular_stakes()` do documento de referência como função helper pura | Lógica já documentada e testável isoladamente |
| Percentuais hardcoded | Constantes em código Python | Tabela `complementares` no PostgreSQL | D-21: percentuais devem ser ajustáveis via UPDATE direto |

**Key insight:** Toda a lógica de validação de complementares é computável a partir do placar string "X-Y" que o parser já extrai. Não é necessário mudar o listener, o parser, ou o schema da tabela `signals`.

## Common Pitfalls

### Pitfall 1: Sinal RED sem placar — complementares ficam pendentes em vez de RED

**What goes wrong:** Código verifica apenas `if placar is None: return None` sem verificar `resultado_principal`.
**Why it happens:** Pensar no fluxo "GREEN com placar" como único caminho de validação.
**How to avoid:** Verificar `resultado_principal` ANTES de verificar `placar`. Se resultado é RED, retornar RED independente de ter placar ou não (D-08).
**Warning signs:** Complementares com `resultado=None` depois de um sinal RED no banco.

### Pitfall 2: Cálculo de stake complementar ignorando tentativa

**What goes wrong:** `stake_comp = stake_inicial * percentual` (sempre T1, sem dobrar).
**Why it happens:** Confundir stake de referência com stake da tentativa atual.
**How to avoid:** Fórmula correta: `stake_comp_TN = stake_inicial * (2^(N-1)) * percentual`. A dobra da complementar é derivada da dobra da principal (D-14).
**Warning signs:** ROI complementares subestimado nos REDs e superestimado nas tentativas altas.

### Pitfall 3: regra empate_3_3_4_4 com falso-positivo

**What goes wrong:** Implementar como apenas `(casa + fora) in (6, 8)` sem verificar `casa == fora`.
**Why it happens:** Lembrar apenas dos totais (6 e 8) sem lembrar que o mercado é sobre placares EXATOS simétricos.
**How to avoid:** A condição correta é `(casa + fora) in (6, 8) AND casa == fora`. Isso exclui 1-5, 5-1, 2-6, 6-2, etc.
**Warning signs:** Um placar 2-4 sendo classificado como GREEN em empate_3_3_4_4.

### Pitfall 4: Seed data duplicado em restarts

**What goes wrong:** `INSERT INTO mercados ... VALUES ...` sem `ON CONFLICT DO NOTHING` levanta `UniqueViolation` no segundo start.
**Why it happens:** Esquecer que `ensure_schema()` é chamado toda vez que o listener ou dashboard inicia.
**How to avoid:** Sempre usar `ON CONFLICT (slug) DO NOTHING` nas tabelas com UNIQUE constraint.
**Warning signs:** `psycopg2.errors.UniqueViolation` nos logs de startup.

### Pitfall 5: Filtro de liga/entrada no card complementares ignora os filtros do dashboard

**What goes wrong:** `calculate_roi_complementares()` recebe sempre todos os sinais, não os filtrados pelo liga/entrada dropdown.
**Why it happens:** Esquecer que o master callback já filtra `history` — basta repassar a mesma lista.
**How to avoid:** No callback, passar a mesma variável `history` (já filtrada) para `calculate_roi_complementares()`.
**Warning signs:** ROI complementares não muda quando um filtro de liga é aplicado.

### Pitfall 6: percentual armazenado como porcentagem (20) em vez de fração (0.20)

**What goes wrong:** `stake_comp = stake_inicial * 20 * (2^(N-1))` resulta em stakes 100x maiores.
**Why it happens:** Ambiguidade entre "20%" como string e 0.20 como NUMERIC.
**How to avoid:** Armazenar como decimal fracionário (`NUMERIC(5,4)` permite até 9.9999). O seed usa 0.20, 0.01, etc. A fórmula multiplica diretamente: `stake_principal * percentual`.
**Warning signs:** Stakes calculadas de R$2.000 em vez de R$20.

### Cálculo de stake por tentativa (do documento de referência)

```python
# Source: /Users/luciano/Downloads/documentacao_entradas_mercado.md — Seção 6
def calcular_stakes(stake_inicial: float, tentativa: int, complementares: list[dict]) -> dict:
    """
    Retorna dict com stake principal e cada complementar para uma tentativa.
    tentativa: 1 a 4
    complementares: lista de dicts com 'slug', 'percentual', 'odd_ref', 'nome_display'
    """
    fator = 2
    stake_principal = stake_inicial * (fator ** (tentativa - 1))

    resultado = {
        "tentativa": tentativa,
        "principal": round(stake_principal, 2),
        "complementares": {},
        "total_complementar": 0.0,
        "total_geral": 0.0,
    }

    for comp in complementares:
        valor = round(stake_principal * comp["percentual"], 2)
        resultado["complementares"][comp["slug"]] = {
            "nome": comp["nome_display"],
            "percentual": comp["percentual"],
            "odd_ref": comp["odd_ref"],
            "valor": valor,
        }
        resultado["total_complementar"] += valor

    resultado["total_complementar"] = round(resultado["total_complementar"], 2)
    resultado["total_geral"] = round(stake_principal + resultado["total_complementar"], 2)
    return resultado
```

## Open Questions

1. **Comportamento do card de complementares quando não há filtro de entrada**
   - O que sabemos: os complementares são diferentes por mercado principal (Over 2.5 tem percentuais diferentes de Ambas Marcam)
   - O que está incerto: ao mostrar "todas as entradas", somar complementares de mercados diferentes pode gerar número sem significado
   - Recomendação: quando entrada=None (sem filtro), exibir o card com mensagem "Selecione uma entrada para ver complementares" em vez de tentar agregar todos os mercados

2. **Comportamento com sinais que têm placar mas resultado=GREEN e total < threshold**
   - O que sabemos: placar vem do sinal GREEN (a principal acertou). Um sinal Over 2.5 GREEN com placar 2-1 deve ter Over 3.5 como RED (total=3, não >3.5)
