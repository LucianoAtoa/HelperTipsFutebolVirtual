# Domain Pitfalls

**Domain:** Telegram user client (Telethon) + betting signal capture + analytics dashboard
**Researched:** 2026-04-02
**Confidence:** HIGH (session/async pitfalls from official docs + GitHub issues), MEDIUM (analytics/statistics pitfalls from community sources)

---

## Critical Pitfalls

Mistakes that cause data loss, account termination, or rewrites.

---

### Pitfall 1: psycopg2 Blocks the Telethon asyncio Event Loop

**What goes wrong:** The project stack uses Telethon (async) with psycopg2 (synchronous). Every INSERT/UPDATE call to PostgreSQL blocks the asyncio event loop. While the database write runs, Telethon cannot process incoming Telegram events. If two signals arrive in rapid succession — which happens regularly in active tipster groups — the second event sits in the queue while the first write completes. Under sustained load, the queue overflows silently and messages are dropped without any error.

**Why it happens:** psycopg2 was not designed for asyncio. It calls blocking C-level socket operations that hold the GIL and suspend the entire event loop, not just the current coroutine.

**Consequences:** Signals are silently dropped. The database record count diverges from the actual Telegram message count. ROI analytics become unreliable because the dataset is incomplete.

**Prevention:** Wrap every psycopg2 call in `asyncio.to_thread()` so it runs in a thread pool without blocking the loop. Alternatively, replace psycopg2-binary with psycopg[binary] (psycopg3) or asyncpg, both of which are natively async. `asyncio.to_thread()` is the lowest-change fix since the rest of the stack stays the same.

```python
# Safe pattern with psycopg2-binary (current stack)
import asyncio

async def save_signal(conn, data):
    await asyncio.to_thread(_blocking_insert, conn, data)

def _blocking_insert(conn, data):
    with conn.cursor() as cur:
        cur.execute("INSERT INTO signals ...", data)
    conn.commit()
```

**Detection:** Add timing logs around DB calls. Any call over 5ms in a single-threaded asyncio app signals a blocking problem.

**Phase:** Must be addressed in Phase 1 (listener + storage foundation). Getting this wrong poisons all downstream analytics.

### Pitfall 2: AuthKeyDuplicatedError from Concurrent Session Usage

**What goes wrong:** The `.session` file generated by Telethon encodes an authorization key tied to a specific IP/connection. If two processes open the same `.session` file simultaneously — even an interactive Python shell while the listener is running — Telegram invalidates the key with `AuthKeyDuplicatedError`. The listener crashes and requires re-authentication via phone + SMS code.

**Why it happens:** Telegram's MTProto protocol detects the same auth key coming from two different connections (or two different IPs) and forces invalidation as a security measure.

**Consequences:** Listener goes down. During re-auth downtime, all signals from the group are missed — there is no backfill mechanism (see Pitfall 3).

**Prevention:**
- Never run two scripts simultaneously using the same session name.
- Never open an interactive Python shell while the listener daemon is running.
- Use a process lock file (e.g., `/tmp/helpertips.lock`) to prevent accidental double-start.
- Set a distinctive session name that makes concurrent use obvious: `helpertips_listener.session`.

**Detection:** `AuthKeyDuplicatedError` in logs is the exact error. Any time the listener crashes on startup, check for an existing process using the same session.

**Phase:** Phase 1 setup. Add the lock file guard before writing any other listener code.

### Pitfall 3: No Recovery for Messages Missed While the Listener Is Offline

**What goes wrong:** When the Telethon client is offline (crashed, restarted, network dropped), Telegram delivers events only to connected clients. When the listener reconnects, it receives new events going forward but not the gap that occurred during downtime. The `catch_up=True` parameter exists but has a documented history of bugs, infinite loops on reconnect, and incomplete gap filling — GitHub issues #746, #1125, #3041, and #4535 all track versions of this problem.

**Why it happens:** Telethon's update gap recovery relies on calling `GetDifference` against Telegram's servers, but this mechanism is fragile in megagroups and broadcast channels. The client may not have a reliable `pts` (update sequence counter) to reference after a clean restart.

**Consequences:** Any signals sent while the listener was down are permanently missing from the database. Analytics will silently undercount signals for those time windows. If the listener crashes during a high-volume session, win-rate statistics for that period are corrupted.

**Prevention:**
- Treat the listener as a process that must not crash. Add `try/except` around all event handlers that logs and recovers rather than propagates.
- On restart, query the Telegram group's message history manually (`client.get_messages()`) for the N messages after the last known `message_id` in the database and backfill them.
- Write the manual backfill as a utility script that runs on every start, not just on crash recovery.

**Detection:** On startup, compare the `message_id` of the last DB record against the latest message in the group. A gap larger than expected indicates missed messages.

**Phase:** Phase 1 (listener reliability). The backfill utility can be Phase 2 polish, but the crash-recovery `try/except` must be Phase 1.

### Pitfall 4: Session File Committed to Git

**What goes wrong:** The `.session` file contains the full authentication state for the Telegram account — anyone who obtains it has complete access to the account: all private messages, groups, contacts. It is functionally equivalent to handing over the account password.

**Why it happens:** Developers focus on committing the working code and forget that the `.session` file is created at runtime in the project root. It is not in the default Python `.gitignore`.

**Consequences:** If the repo is ever made public or shared, the Telegram account is fully compromised. Telegram does not expire sessions on inactivity by default.

**Prevention:**
- Add `*.session` and `*.session-journal` to `.gitignore` before the first commit.
- Also add `.env` containing `TELEGRAM_API_ID` and `TELEGRAM_API_HASH`.
- Run `git status` check in startup script: if `.session` file is tracked by git, abort with an error message.

**Detection:** `git ls-files | grep .session` returns output — this means the file is tracked.

**Phase:** Pre-Phase 1, during project initialization. Non-negotiable before first `git push`.

### Pitfall 5: Regex Parser Breaks Silently When Message Format Changes

**What goes wrong:** The tipster group may change message formatting at any time — different emojis, different field order, new result notation, reformatted odds. The regex parser either fails to match (signal not stored at all) or matches partially (corrupt record stored with `None` fields). Because the listener runs unattended, neither failure mode raises an alert.

**Why it happens:** Regex-based parsing against natural-language Telegram messages is inherently brittle. There is no schema contract between the tipster and the consumer.

**Consequences:** Silent data corruption. Days or weeks of signals may be missing or corrupt before the user notices through dashboard anomalies. Historical analysis becomes unreliable.

**Prevention:**
- Log every message that arrives from the group regardless of whether parsing succeeds.
- Log a distinct warning when a message does not match the expected pattern.
- Implement a raw message storage column (`raw_text TEXT`) in the signals table so unparsed messages can always be re-processed later after a regex fix.
- Keep a "parse failure counter" queryable from the dashboard.

**Detection:** The raw message log grows but the `signals` table count does not increase proportionally. Parse failure counter is non-zero.

**Phase:** Phase 1 (parser + storage). The `raw_text` column costs nothing and prevents irreversible data loss.

## Moderate Pitfalls

---

### Pitfall 6: ROI Simulation With Insufficient Sample Size Produces Misleading Conclusions

**What goes wrong:** The dashboard shows ROI statistics after only 50-100 signals. The user reads "Liga X has 73% GREEN rate" and adjusts betting strategy. With small samples, this number is noise. A confidence interval around a 73% rate from 50 samples spans roughly 60–84%. The apparent pattern is meaningless.

**Why it happens:** The dashboard surfaces numbers as soon as they exist. There is no indication that 50 samples is far below the 300–500 minimum needed for statistically meaningful rates in sports betting analytics.

**Consequences:** The user makes real money decisions based on spurious patterns. A "hot streak" in the data looks like a real edge. When the variance corrects, the strategy underperforms.

**Prevention:**
- Display a sample size count next to every percentage in the dashboard.
- Show a visual warning (e.g., greyed-out stat, asterisk) when a filter combination produces fewer than 100 signals.
- Document in the dashboard that actionable patterns require 300+ signals per segment.

**Detection:** Filter by a specific league/time combination and check the signal count. If under 100, confidence is low.

**Phase:** Phase 2 (dashboard). A sample-size annotation is a small addition to every stat card.

### Pitfall 7: Duplicate Signals from Edit Events Processed as New Signals

**What goes wrong:** The group workflow is: (1) post the signal as a new message, (2) edit that same message to add GREEN/RED result. Telethon fires both `events.NewMessage` and `events.MessageEdited`. If the handler for `NewMessage` stores the signal and the handler for `MessageEdited` stores it again without upsert logic, the signal appears twice in the database. ROI and win-rate calculations double-count results.

**Why it happens:** Treating `NewMessage` and `MessageEdited` as independent events that both INSERT records, rather than modeling them as "initial signal" and "result update" on the same record.

**Prevention:**
- Use `ON CONFLICT (message_id) DO UPDATE` (upsert) as the only write path.
- The composite key should be `(chat_id, message_id)` — message IDs are unique per channel but not across channels.
- `NewMessage` stores the initial signal with `result = NULL`.
- `MessageEdited` updates the existing record to set `result = 'GREEN'/'RED'` and `final_score`.

**Detection:** `SELECT message_id, COUNT(*) FROM signals GROUP BY message_id HAVING COUNT(*) > 1` returns rows.

**Phase:** Phase 1 (data model + event handlers). The upsert pattern must be designed in from the first schema migration.

### Pitfall 8: Overfitting the Analytics to Current Message Format

**What goes wrong:** The dashboard is built assuming specific field names, specific league names, specific entry types that the parser extracts today. When the tipster group evolves their format (new leagues added, entries renamed, emojis replaced), the parser extracts different values and the dashboard filters break because they reference hardcoded strings.

**Prevention:**
- Store raw league names and entry types as extracted strings. Do not normalize to a hardcoded enum.
- Build all dashboard filters dynamically from the actual distinct values in the database (`SELECT DISTINCT liga FROM signals`).
- Validate parser output against a known-good test set of saved messages and alert on divergence.

**Phase:** Phase 1 (schema) and Phase 2 (dashboard). Dynamic filters are straightforward if the data model is flexible from the start.

## Minor Pitfalls

---

### Pitfall 9: Telegram Account Ban from Unusual API Activity

**What goes wrong:** Telegram's anti-spam systems flag accounts that make rapid or unusual API calls. Telethon used for legitimate passive listening is very low-risk, but calling APIs in bulk (e.g., fetching all message history in a loop on every restart) or running from a newly created account elevates risk.

**Prevention:**
- Use a well-established Telegram account (not newly created).
- Avoid real phone numbers from high-risk regions for this API credential.
- Do not batch-request message history aggressively on startup — rate-limit backfill operations with `asyncio.sleep(0.5)` between API calls.
- Handle `FloodWaitError` by sleeping the exact number of seconds specified in `error.seconds`.

**Phase:** Phase 1 (listener). Add `FloodWaitError` handling before going to production.

---

### Pitfall 10: Connection Pool Exhaustion Under Dashboard + Listener Concurrency

**What goes wrong:** The listener writes to PostgreSQL continuously. If the web dashboard also opens database connections (e.g., one per HTTP request), the total connections can exceed PostgreSQL's default limit (100) or saturate available file descriptors. Under load, new connections start failing.

**Prevention:**
- Use a connection pool (e.g., `psycopg2.pool.ThreadedConnectionPool` or pgBouncer).
- Set a `max_connections` cap in the pool lower than the PostgreSQL server limit.
- The listener should own a single long-lived connection or a pool of 2–3 max.

**Phase:** Phase 2 (when dashboard is added). Phase 1 with only the listener is unlikely to hit this.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| Phase 1: Listener + DB setup | psycopg2 blocks asyncio event loop (Pitfall 1) | Use `asyncio.to_thread()` from day one |
| Phase 1: Session initialization | `.session` file committed to git (Pitfall 4) | `.gitignore` entry before first commit |
| Phase 1: Event handlers | Duplicate records from edit events (Pitfall 7) | Upsert-only write path with composite key |
| Phase 1: Message parser | Silent parse failures corrupt dataset (Pitfall 5) | Store `raw_text`; log unmatched messages |
| Phase 1: Process management | Session invalidated by concurrent process (Pitfall 2) | Process lock file + single session name |
| Phase 1: Listener reliability | Signals missed while offline, no recovery (Pitfall 3) | Manual backfill on startup from last known ID |
| Phase 2: Dashboard stats | Small-sample statistics look actionable (Pitfall 6) | Sample count + confidence warning on every stat |
| Phase 2: Dashboard filters | Hardcoded strings break on format changes (Pitfall 8) | Dynamic filters from DB distinct values |
| Phase 2+: Concurrent DB access | Connection exhaustion (Pitfall 10) | Connection pool capped below PG limit |
